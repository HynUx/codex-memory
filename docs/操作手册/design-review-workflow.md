# 设计审查工作流

> 最后更新: 2026-07-04
> 适用范围: 所有涉及系统/架构/方案设计的工作

---

## 一、设计必须遵循的顺序

任何设计必须按以下逻辑链条展开，跳过任意一步即为不合格：

```
Step 1: 使用者视角 → 谁在用、怎么用、在什么场景用
Step 2: 交互协议     → 指令/CLI/API 的具体调用方式
Step 3: 运行逻辑     → 每一步操作的前置条件、后置结果、错误路径
Step 4: 存储设计     → 数据结构、索引、持久化
Step 5: 进化机制     → 数据如何增长、合并、清理、纠错
Step 6: 迁移路径     → 从旧系统到新系统的完整转换方案
```

**禁止**直接进入 Step 4/5 开始画表和写 schema。必须先回答"谁用它、怎么用"。

---

## 二、设计自审清单（提交子 agent 审查前必须自检）

在发起子 agent 审查之前，设计者必须逐条确认以下问题已经回答：

### 使用者视角

- [ ] 这个系统的直接使用者是谁？（人还是 agent？）
- [ ] 使用者需要做什么操作？每一步的输入和输出是什么？
- [ ] 如果使用者是 agent，agent 怎么知道何时该调用？决策树画出来了吗？

### 交互协议

- [ ] 每个 CLI 命令/API 的参数、返回值、错误码是否完整定义？
- [ ] 输出格式是否对使用者友好（agent 能 parse？人能读懂？）
- [ ] 错误输出是否给了明确的修复方向？

### 运行逻辑

- [ ] 每个操作在 **正常路径** 下的执行步骤是什么？
- [ ] 每个操作在 **异常路径**（文件损坏、并发冲突、权限错误、磁盘满）下的行为是什么？
- [ ] 多个操作并发执行时，数据一致性如何保证？
- [ ] 如果某一步失败，整个系统的状态是可恢复的吗？

### 存储设计

- [ ] schema 中的每个字段是否有明确的业务含义和取值范围？
- [ ] 约束条件是否在数据库层面强制执行（CHECK、UNIQUE、FOREIGN KEY）？
- [ ] 索引是否同步更新（trigger 或应用层保证）？

### 进化机制

- [ ] 每条数据进入系统后，它的生命周期是怎样的？（创建 → 检索 → 合并 → 归档/删除）
- [ ] 合并/清理的触发条件是参数化的还是硬编码的？参数有数据支撑吗？
- [ ] 错误的、过时的、矛盾的数据如何处理？

### 迁移路径

- [ ] 旧数据到新数据的转换方案是否无损？
- [ ] 是否有回滚方案？
- [ ] 转换过程中系统是否可用？

## 三、子 agent 审查流程

### 触发条件

以下场景**必须**触发子 agent 审查：

1. 所有涉及系统/架构/方案设计的文档（spec、design doc），在发给用户 review 之前
2. 调研报告完成后、设计开始前（可选）
3. 用户明确要求 review 的场景

### 执行参数

```
模型:          deepseek-v4-pro
推理强度:      high
审查立场:      否定式批判（找问题，不找优点）
输出格式:      每个问题标明 致命/重要/建议，附带具体场景和修复方向
```

### 审查范围

子 agent 必须覆盖以下 10 个维度（不全即审查结果不合格，需重新提交）：

| # | 维度 | 核心追问 |
|---|------|----------|
| 1 | 完整性 | 哪些关键设计完全缺失？ |
| 2 | 并发与一致性 | 多实例同时操作时可能出什么问题？ |
| 3 | 数据完整性 | 文件损坏、不同步时恢复路径是什么？ |
| 4 | 性能 | 数据量增长后哪些操作会退化？ |
| 5 | 安全与隐私 | 敏感信息可能被记录/泄露吗？ |
| 6 | 可维护性 | 配置、schema、迁移路径是否清晰可扩展？ |
| 7 | 假设验证 | 哪些"想当然"的假设没有数据支撑？ |
| 8 | 错误恢复 | 每个操作失败时的行为是否定义？ |
| 9 | 接口设计 | CLI/API 是否完整、语义是否清晰、参数是否合理？ |
| 10 | 用户体验 | 使用者是否知道如何用、用得顺畅？ |

### 审查通过标准

**通过条件**（缺一不可）：
- 没有"致命"级别的未解决问题
- "重要"级别的问题不超过 3 个
- 所有"重要性"问题的修复方向已明确

**不通过时**：设计者根据审查报告重做设计，再提交新审查。直到通过为止。

## 四、设计链条一致性检查

设计的各个部分必须构成一条自洽的逻辑链：

```
使用者需求
    ↓ 决定了
交互协议（怎么操作、有哪些命令）
    ↓ 决定了
运行逻辑（每个操作的执行路径、错误路径）
    ↓ 决定了
存储与索引（数据怎么存、怎么查）
    ↓ 决定了
进化与清理（数据怎么生长）
```

**检查方法**：从后往前追溯。选任意一个表字段或命令参数，追问"这个字段/参数对应的使用者需求是什么？"——如果回答不了，这个字段/参数就是多余的。

| 文档类型 | 归档位置 | 命名规范 |
|----------|----------|----------|
| 调研报告 | `docs/调研文档/` | `YYYY-MM-DD-<主题>-research.md` |
| 设计文档 | `docs/设计文档/` | `YYYY-MM-DD-<主题>-design.md` |
| 实施计划 | `docs/计划文档/` | `YYYY-MM-DD-<主题>-plan.md` |

| superpowers 产出 | `docs/superpowers/` | 按插件自身规范 |

## 六、工作流总结图

```
用户提出需求
    │
    ├──→ 调研（搜索、分析、对比）
    │       │
    │       └──→ 调研报告 → docs/调研文档/
    │
    ├──→ 设计（按逻辑链顺序：使用者→协议→逻辑→存储→进化→迁移）
    │       │
    │       ├──→ 设计者自审（检查清单）
    │       │
    │       └──→ 子 agent 审查 (deepseek-v4-pro, high, 否定式)
    │               │
    │               ├── 通过 → 发给用户 review
    │               │
    │               └── 不通过 → 重做设计 → 再审查
    │
    ├──→ 用户 review
    │       │
    │       ├── 通过 → 进入实施
    │       │
    │       └── 不通过 → 返工
    │
    └──→ 实施（按计划文档执行）
```

## 七、审查的核心原则

子 agent 审查必须从**第一性原理**出发，以以下目标为审查标准：

| 原则 | 含义 | 审查时追问 |
|------|------|------------|
| **简洁** | 最简单的方案就是最好的方案 | 这个字段/命令/逻辑是必要的吗？去掉会怎样？ |
| **可靠** | 任何操作失败时系统状态可恢复 | 如果这一步出错，数据是脏的还是干净的？怎么修？ |
| **泛化能力强** | 设计不绑定特定场景 | 换一个项目、换一个模型、换一种用法，设计还能工作吗？ |
| **逻辑清晰** | 每个决策的原因可追溯 | 为什么选这个值而不选另一个？有数据支撑还是拍脑袋？ |
| **框架设计优化清晰** | 关注点分离、职责单一 | 这个模块的职责是什么？它的边界在哪里？ |

## 八、发起审查的必须要素

每次发起子 agent 审查时，必须同时提供以下三要素：

### 1. 完整的需求列表

```
| ID | 需求描述 | 优先级(🔴/🟡/🟢) |
```

逐条列出设计要满足的所有需求，每条标注优先级。审查的核心任务是：确认设计是否满足每条需求。

### 2. 明确的审查标准

```
审查优先级（从高到低）:
  1. 需求满足度 — 每条需求是否满足？给出通过/不通过的理由
  2. 工程质量 — 在满足需求的基础上评估简洁/可靠/可维护/可扩展
  
输出格式:
  需求逐条评审: R1-RN 每条 ✅/❌ + 理由
  工程质量问题: 致命/重要/建议 分级 + 修复方向
  总结判断: 通过 / 条件通过 / 不通过
```

### 3. 本轮审查重点

本次审查必须解决的关键问题或待确认的决策点。

### 禁止

- ❌ 不给需求列表就发起审查
- ❌ 审查标准模糊（如只说"看看有没有问题"）
- ❌ 审查产出不包含需求逐条评审
